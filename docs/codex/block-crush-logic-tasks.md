# Задачи для Codex: логика игры Block Crush

## Контекст
Нужно реализовать базовую игровую логику для пазла в стиле **Block Puzzle / Woodoku**.

Игровое поле: квадратная сетка 8x8 или 9x9.
Игроку предлагаются 3 фигуры.
Игрок размещает фигуры на поле.
Если после размещения полностью заполнен ряд или столбец — он очищается.
За размещение и очистку начисляются очки.
Игра заканчивается, если ни одну из доступных фигур больше нельзя поставить на поле.

Важно: сейчас нужен именно **слой логики**, а не финальный polished UI.
Код должен быть чисто разделён на доменную модель, состояние игры и операции над полем.

---

## Задача 1. Доменная модель поля и фигур
**Цель:** создать основу данных для игры.

### Что сделать
- Реализовать модель игрового поля:
  - ширина
  - высота
  - состояние клеток (пусто / занято)
- Реализовать модель фигуры:
  - список локальных клеток фигуры
  - размеры bounding box
  - идентификатор фигуры
- Подготовить набор базовых фигур:
  - 1 клетка
  - 2 клетки
  - 3 клетки в линию
  - квадрат 2x2
  - L-образные
  - T-образные
  - длинные линии
- Добавить фабрику/каталог фигур для выдачи случайных шаблонов.

### Критерии готовности
- Поле можно создать с заданным размером.
- Фигуры описаны как данные, без хардкода в UI.
- Есть единая точка, из которой можно получить список доступных шаблонов фигур.

---

## Задача 2. Проверка возможности размещения фигуры
**Цель:** реализовать валидацию установки фигуры на поле.

### Что сделать
- Добавить метод проверки, можно ли поставить фигуру в указанную клетку якоря.
- Проверять:
  - все клетки фигуры попадают в границы поля
  - ни одна из целевых клеток не занята
- Возвращать понятный результат проверки:
  - success / fail
  - причина ошибки (выход за границы / занято)

### Критерии готовности
- Для любой фигуры можно проверить валидность размещения.
- Проверка не меняет состояние поля.
- Логика покрыта базовыми unit-тестами.

---

## Задача 3. Размещение фигуры на поле
**Цель:** применить фигуру к полю.

### Что сделать
- Реализовать метод размещения фигуры на поле после успешной проверки.
- Помечать соответствующие клетки как занятые.
- Возвращать структуру результата размещения:
  - какие клетки заняты
  - какие ряды очищены
  - какие столбцы очищены
  - сколько очков начислено за ход

### Критерии готовности
- После размещения поле корректно обновляется.
- Данные о результате хода можно использовать отдельно в UI/анимациях.
- Размещение невозможно без успешной предварительной валидации.

---

## Задача 4. Очистка заполненных рядов и столбцов
**Цель:** реализовать основную механику Block Crush.

### Что сделать
- После каждого успешного размещения проверять:
  - полностью заполненные ряды
  - полностью заполненные столбцы
- Очищать их за один шаг.
- Корректно обрабатывать случай, когда за один ход очищаются:
  - несколько рядов
  - несколько столбцов
  - и ряды, и столбцы одновременно
- Возвращать список очищенных клеток.

### Критерии готовности
- Полностью заполнённые линии действительно очищаются.
- Пересекающиеся линии обрабатываются корректно.
- Поведение детерминировано и покрыто тестами.

---

## Задача 5. Система очков
**Цель:** добавить базовый scoring.

### Базовые правила
- Очки за размещение блока: +1 за каждую занятую клетку фигуры.
- Дополнительные очки за очистку:
  - +10 за каждую очищенную линию
- Комбо:
  - если за один ход очищено 2 и более линий, добавить бонус
  - стартовый вариант: `bonus = clearedLines * 5`

### Что сделать
- Реализовать отдельный scorer/service для расчёта очков.
- Не смешивать подсчёт очков с кодом UI.
- Возвращать подробную разбивку очков за ход.

### Критерии готовности
- Очки считаются одинаково для одинаковых входных данных.
- Разбивка очков прозрачна и пригодна для отображения игроку.

---

## Задача 6. Набор из 3 текущих фигур
**Цель:** реализовать стандартный игровой цикл выбора фигур.

### Что сделать
- Игра должна одновременно хранить 3 доступные фигуры.
- После использования одной фигуры она удаляется из набора.
- Когда все 3 фигуры использованы, генерируется новый набор из 3 фигур.
- Добавить сервис генерации набора фигур.
- На текущем этапе допустима случайная генерация без сложного баланса.

### Критерии готовности
- Игрок всегда имеет до 3 доступных фигур.
- Новый набор появляется только после полного расходования предыдущего.
- Логика не зависит от UI.

---

## Задача 7. Проверка состояния Game Over
**Цель:** завершать игру, когда ни одна фигура больше не помещается.

### Что сделать
- Реализовать метод проверки: существует ли хотя бы одно допустимое размещение хотя бы для одной из текущих фигур.
- Проверять все фигуры из текущего набора по всем координатам поля.
- Если ни одного допустимого хода нет — игра переходит в состояние поражения.

### Критерии готовности
- Game Over определяется корректно.
- Проверка работает для любого состояния поля.
- Логика покрыта тестами на частично и почти полностью заполненное поле.

---

## Задача 8. Главный игровой state/controller
**Цель:** собрать всё в единый сценарий хода.

### Что сделать
- Создать основной `GameState` / `GameController`, который управляет:
  - полем
  - текущими 3 фигурами
  - очками
  - состоянием игры (active / game over)
- Реализовать сценарий хода:
  1. проверить размещение
  2. поставить фигуру
  3. очистить линии
  4. начислить очки
  5. удалить использованную фигуру из набора
  6. при необходимости сгенерировать новый набор
  7. проверить game over
- Возвращать результат хода в виде отдельной структуры.

### Критерии готовности
- Один публичный метод обрабатывает полный игровой ход.
- После хода состояние игры консистентно.
- Код удобно расширять дальше.

---

## Задача 9. Unit-тесты на ключевые сценарии
**Цель:** зафиксировать корректность логики.

### Минимальный набор тестов
- фигура ставится в пустое место
- фигура не ставится за границы
- фигура не ставится на занятые клетки
- очищается один ряд
- очищается один столбец
- очищаются несколько линий за один ход
- правильно считаются очки
- после использования 3 фигур генерируется новый набор
- корректно определяется game over

### Критерии готовности
- Есть edit mode / unit tests для всей базовой логики.
- Тесты не зависят от сцены и UI.

---

## Требования к архитектуре
- Не смешивать UI и игровую логику.
- Логика должна быть тестируемой без сцены.
- По возможности использовать простые immutable/result-структуры для ответа операций.
- В коде избегать монолитного класса на всю игру.

Рекомендуемое разбиение:
- `Board`
- `BlockShape`
- `BlockCatalog`
- `PlacementValidator`
- `LineClearResolver`
- `ScoreCalculator`
- `FigureBag` / `PieceSetGenerator`
- `GameOverChecker`
- `GameStateController`
